# KB README Index - Оглавление базы знаний

## Правила ведения KB

1. **Единая точка входа**
   - Файл `KB/README_INDEX.md` является единственным оглавлением базы знаний.
   - Каждый файл `README_*.md` в папке `KB/` должен быть перечислен в одной из секций индекса.
   - В таблицах секций присутствуют все четыре колонки (**Файл**, **Описание**, **Ключевые темы**, **Актуализация**); колонка «Актуализация» может оставаться пустой (см. п.3).

2. **Именование файлов**
   - Документация оформляется в виде файлов `README_<ТЕМА>.md` (например, `README_LOADING_FILES.md`, `README_DB-VECTOR.md`, `README_links.md`).
   - Исторические исключения (например, `langchain-pg_README_REST.md`) допускаются, но для новых файлов используется общая схема `README_<ТЕМА>.md`.
   - Схема базы данных хранится в одном каноническом файле `current_DB_schema.sql`; при изменениях схемы он обновляется и при необходимости упоминается в индексе.

3. **Актуализация (колонка дат)**
   - В колонке **Актуализация** используется формат даты `YYYY-MM-DD` (например, `2026-02-11`).
   - Дата в колонке **Актуализация** обновляется при содержательных изменениях документа (новые разделы, изменённая логика, новые сущности), а не при правке опечаток.

4. **Структура и содержание документов**
   - В начале каждого README должны быть заголовок и краткое описание назначения документа (1–2 предложения).
   - Содержание README должно соответствовать «Ключевым темам», указанным для него в индексе.
   - В конце документа рекомендуется добавлять блок «Связанная документация» со ссылками на другие README, если такие связи есть.

5. **Ссылки и навигация**
   - Внутри KB используются относительные ссылки вида `[README_REST.md](./README_REST.md)`.
   - На код и тесты ссылаются относительными путями от корня репозитория (например, `routes/ai.js`, `tests/test_rag_retrieval.js`).
   - При переименовании или удалении README необходимо обновить все ссылки на него и соответствующую запись в индексе.

6. **Синхронизация с кодом и конфигурацией**
   - При добавлении нового loader’а, эндпоинта, таблицы, поля БД или другой значимой сущности:
     - обновляется соответствующий README (или создаётся новый);
     - добавляется или обновляется запись в `README_INDEX.md` (включая колонку **Актуализация**);
     - при изменении схемы БД обновляется `current_DB_schema.sql` и при необходимости `README_DB-VECTOR.md`.

7. **Секции индекса**
   - По умолчанию новый документ размещается в одной из существующих секций:
     1. Архитектура и технологии  
     2. База данных  
     3. API и интеграции  
     4. Загрузка и обработка файлов  
     5. AI Items и метаданные  
     6. Тестирование  
     7. Утилиты и прочее  
   - Если тема плохо вписывается в существующие секции, её временно размещают в «Утилиты и прочее» и при следующем рефакторинге индекса решают, нужна ли для неё отдельная секция.

8. **Блок «Быстрый поиск по темам»**
   - В блоке «Быстрый поиск по темам» перечисляются только ключевые сценарии (например, «Начать изучение проекта», «Работа с файлами и pipeline», «AI Items и связи», «Конфигурация», «Интеграции»).
   - В этот блок добавляются только наиболее важные README как точки входа, без полного дублирования всех документов из таблиц.

9. **Язык и стиль**
   - Основной язык KB — русский; названия сущностей, API, таблиц и полей пишутся так же, как в коде (например, `AI Item`, `chunk`, `Step1`, `Step2`).
   - Для структурированных перечислений используются таблицы; для алгоритмов и сценариев — маркированные и нумерованные списки.
   - Стиль документов должен оставаться единообразным: краткие формулировки, техническая точность и опора на реальные сущности системы.

10. **Чеклист перед коммитом (рекомендация)**
    - Если в коммите есть изменения в KB:
      - проверить, что соответствующие файлы `README_*.md` добавлены или обновлены в `README_INDEX.md`;
      - проверить, что значения в колонке **Актуализация** обновлены там, где были содержательные изменения;
      - убедиться, что все ссылки из индекса и изменённых README указывают на существующие файлы.
